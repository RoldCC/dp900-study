Summary

Data is classified as structured (tables with fixed schema), semi-structured (JSON, XML, YAML — self-describing but flexible), or unstructured (images, video, documents — no defined schema).

  • Structured → relational databases, spreadsheets
  • Semi-structured → JSON APIs, log files, NoSQL stores
  • Unstructured → blob/object storage, data lakes
⚠ Exam tip: Semi-structured data is NOT unstructured — it has an implied structure via tags or key-value pairs.

Common file formats serve different purposes. CSV is row-based and human-readable. JSON and XML are hierarchical. Parquet and Avro are binary columnar/row formats optimised for analytics. Delta Lake adds ACID transactions on top of Parquet files.

  • Parquet: columnar, high compression, fast analytical queries
  • Avro: row-based binary, good for streaming (schema in header)
  • ORC: columnar, used in Hive/Hadoop ecosystems
  • Delta Lake: Parquet + transaction log (ACID on data lakes)
⚠ Exam tip: Parquet is columnar — ideal for reading a few columns from millions of rows. CSV/JSON/Avro are row-based.

Relational databases store data in normalised tables linked by foreign keys, enforcing a fixed schema and supporting SQL. Non-relational (NoSQL) databases trade schema rigidity for flexibility and horizontal scale, using key-value, document, column-family, or graph models.

  • Relational: strong consistency, ACID, joins, SQL
  • Key-value: ultra-fast lookups by primary key (e.g. Azure Table Storage)
  • Document: JSON/BSON documents, flexible schema (e.g. Cosmos DB NoSQL API)
  • Column-family: wide rows, sparse data (e.g. Cosmos DB Cassandra API)
  • Graph: entities as nodes, relationships as edges (e.g. Cosmos DB Gremlin API)
⚠ Exam tip: Know which NoSQL model fits which use case — graph for social networks, key-value for caches, document for catalogues.

OLTP systems handle high-volume, short-duration transactions — inserts, updates, deletes. They enforce ACID properties to guarantee data integrity even with concurrent users.

  • Atomicity — all operations in a transaction succeed or all fail
  • Consistency — data always moves between valid states
  • Isolation — concurrent transactions don't interfere
  • Durability — committed data survives failures
⚠ Exam tip: OLTP = row-oriented, many small transactions, normalised schema. Optimise for write speed and concurrency.

OLAP systems are optimised for complex queries over large historical datasets, supporting business intelligence and reporting. Data is loaded via ETL/ELT pipelines from OLTP sources into data warehouses or data lakes.

  • ETL: Extract → Transform → Load (transform before loading)
  • ELT: Extract → Load → Transform (transform in-place, common with cloud DWs)
  • Data warehouse: structured, schema-on-write, fast BI queries
  • Data lake: raw data (any format), schema-on-read, flexible
  • Data lakehouse: combines lake storage with warehouse query performance
⚠ Exam tip: OLAP ≠ OLTP. OLAP reads large datasets for analysis; OLTP writes small transactions for operations.

Batch processing collects and processes data in scheduled groups (high latency, high throughput). Stream processing processes data continuously as it arrives (low latency, event-by-event or micro-batch).

  • Batch: nightly ETL jobs, billing runs, monthly reports
  • Stream: fraud detection, IoT telemetry, real-time dashboards
  • Lambda architecture: combines both layers
⚠ Exam tip: Azure Stream Analytics processes streaming data in real-time. Azure Data Factory is batch-oriented.

    Summary

    Relational databases store data in tables (relations) composed of rows and columns. Tables are linked by foreign keys referencing primary keys in other tables. Normalisation reduces redundancy by organising data into related tables.

    • Primary key: uniquely identifies each row
    • Foreign key: links rows between tables
    • Index: speeds up queries at the cost of write overhead
    • View: saved query treated as a virtual table
    ⚠ Exam tip: Normalisation removes duplicate data. Denormalisation (e.g. in star schema) trades redundancy for query speed.

    SQL is divided into statement groups based on what they do: DDL defines structure, DML manipulates data, DCL controls permissions.

    • CREATE, ALTER, DROP → DDL (Data Definition Language)
    • SELECT, INSERT, UPDATE, DELETE → DML (Data Manipulation Language)
    • GRANT, REVOKE, DENY → DCL (Data Control Language)
    • BEGIN TRAN, COMMIT, ROLLBACK → TCL (Transaction Control)
    ⚠ Exam tip: DROP TABLE is DDL — it removes the table definition. DELETE is DML — it removes rows but keeps the table.

    Azure SQL Database is a fully managed PaaS database engine built on SQL Server. Microsoft handles patching, backups, and high availability. It supports serverless compute and elastic pools for cost optimisation.

    • Deployment: Single Database or Elastic Pool
    • Compute: Provisioned or Serverless (auto-pause)
    • 99.99% SLA, built-in geo-redundant backups
    • Hyperscale tier: up to 100 TB, fast scaling
    ⚠ Exam tip: Azure SQL Database is PaaS — you manage data and schema, Azure manages the OS, patches, and hardware.

    SQL Managed Instance is a fully managed PaaS service that provides near-100% compatibility with on-premises SQL Server. It supports features SQL Database lacks: SQL Server Agent, cross-database queries, Service Broker, linked servers.

    • Best for: lift-and-shift migrations from on-premises SQL Server
    • Deployed in a VNet — private network access
    • Supports SQL Server Agent jobs
    • Instance-scoped features unavailable in Azure SQL Database
    ⚠ Exam tip: If the scenario involves migrating an existing SQL Server with minimal code changes, the answer is usually SQL Managed Instance.

    SQL Server on Azure VMs is IaaS — you run SQL Server on a virtual machine you control. Full OS and SQL Server access. Microsoft provides automated backup and patching assistance but you own the infrastructure.

    • Full control: OS, SQL Server version, custom configurations
    • Supports all SQL Server features (including legacy)
    • Best for: apps needing OS-level access or specific SQL Server versions
    • You manage: patching, backups, HA, scaling
    ⚠ Exam tip: IaaS = you manage the VM. PaaS (SQL DB, MI) = Azure manages the infrastructure. SaaS = everything managed for you.

      Summary

      Azure Blob Storage is object storage for unstructured data at massive scale. Blobs are organised into containers inside storage accounts. Three blob types exist for different write patterns.

      • Block blob: text and binary files (most common)
      • Append blob: optimised for append-only operations (logs)
      • Page blob: random read/write, used for Azure VM disks
      • Access tiers: Hot → frequent, Cool → infrequent, Cold → rare, Archive → offline
      ⚠ Exam tip: Archive tier has the lowest storage cost but highest retrieval cost and latency (rehydration needed).

      Azure Files provides fully managed cloud file shares accessible via SMB (Windows) and NFS (Linux/macOS) protocols. It replaces or supplements on-premises file servers.

      • Mount as a network drive on Windows, Linux, macOS
      • Azure File Sync: sync on-premises servers with cloud shares
      • Use case: shared configuration files, lift-and-shift of apps needing a file share
      ⚠ Exam tip: Azure Files = file shares (SMB/NFS). Azure Blob = object storage. They are different services for different use cases.

      Azure Table Storage is a NoSQL key-value store for semi-structured data. Each row is identified by a composite key: PartitionKey (for distribution) + RowKey (unique within partition). Rows in the same partition can have different columns.

      • Schema-less: each row can have different columns
      • PartitionKey: controls data distribution and query performance
      • RowKey: unique identifier within a partition
      • Low cost, high scale — good for non-relational structured data
      ⚠ Exam tip: Table Storage is NOT a relational database. No JOINs, no foreign keys, no SQL. Query by PartitionKey + RowKey for best performance.

      Azure Cosmos DB is a globally distributed, multi-model NoSQL database service. It offers single-digit millisecond latency, 99.999% SLA, and automatic multi-region replication. Data is stored in containers within databases within Cosmos DB accounts.

      • Resource model: Account → Database → Container → Items
      • Partition key determines data distribution across physical partitions
      • Throughput: measured in Request Units (RU/s)
      • 5 consistency levels: Strong, Bounded Staleness, Session, Consistent Prefix, Eventual
      ⚠ Exam tip: Cosmos DB is multi-model — one account, multiple APIs. You pick the API at database creation time; it cannot be changed.

      Cosmos DB offers five APIs, each mapping to a different data model. The API you choose determines the data model and query language.

      • NoSQL API: JSON documents, proprietary SQL dialect — native Cosmos DB
      • MongoDB API: document model, BSON, compatible with MongoDB drivers
      • Table API: key-value, drop-in replacement for Azure Table Storage
      • Cassandra API: column-family, wide-column, CQL compatible
      • Gremlin API: graph model, vertices and edges, Gremlin traversal language
      ⚠ Exam tip: Use Gremlin for social networks/recommendations (relationships). Use Table API to migrate from Azure Table Storage with minimal code changes.

      Azure Storage replicates data to protect against hardware failures and outages. Redundancy options trade cost for geographic protection.

      • LRS (Locally Redundant): 3 copies in one datacenter — cheapest
      • ZRS (Zone Redundant): 3 copies across 3 availability zones in one region
      • GRS (Geo-Redundant): LRS in primary + async copy to secondary region
      • GZRS (Geo-Zone Redundant): ZRS in primary + async copy to secondary region — most resilient
      ⚠ Exam tip: RA-GRS and RA-GZRS add read access to the secondary region. Without RA-, secondary is only accessible during failover.

        Summary

        A modern data warehouse ingests data from multiple sources, stores it in a data lake, transforms it, and loads it into an analytical store for reporting. The medallion architecture organises lake data into Bronze (raw), Silver (cleaned), and Gold (aggregated) layers.

        • Bronze: raw ingestion, no transformation
        • Silver: cleaned, deduplicated, conformed
        • Gold: business-ready aggregates for BI
        ⚠ Exam tip: The medallion architecture (Bronze/Silver/Gold) is associated with Azure Databricks and Delta Lake, not Azure Synapse specifically.

        Azure Data Factory is a cloud ETL/ELT service for orchestrating data movement and transformation pipelines. It connects to 90+ data sources using linked services and connectors.

        • Pipeline: logical grouping of activities
        • Activity: a single step (Copy, Mapping Data Flow, HDInsight, etc.)
        • Linked service: connection string to a data store or compute
        • Dataset: named reference to data within a linked service
        • Integration runtime: compute infrastructure for pipeline execution
        ⚠ Exam tip: ADF is an orchestration service — it moves and transforms data. It is NOT a storage service or query engine.

        ADLS Gen2 is Azure Blob Storage with the Hierarchical Namespace (HNS) feature enabled. HNS gives true directory semantics, enabling atomic rename/delete on directories and fine-grained POSIX ACLs on files and folders.

        • Built on Blob Storage — same low cost, same access tiers
        • Hierarchical namespace: true folders, not just path prefixes
        • POSIX ACLs for fine-grained access control
        • Optimised for analytics workloads (Hadoop, Spark, Synapse)
        ⚠ Exam tip: ADLS Gen2 = Blob Storage + Hierarchical Namespace. If you need a data lake for analytics, enable HNS on a Blob Storage account.

        Azure Synapse Analytics is an integrated analytics platform combining data warehousing, big data, and data integration in one service. It includes a dedicated SQL pool (formerly SQL Data Warehouse), serverless SQL, Spark pools, Synapse Pipelines (ADF-compatible), and Synapse Link.

        • Dedicated SQL pool: MPP data warehouse, massively parallel
        • Serverless SQL pool: query data lake files without loading (pay per query)
        • Apache Spark pool: notebook-based data engineering and ML
        • Synapse Link: near-real-time analytics on Cosmos DB operational data
        ⚠ Exam tip: Synapse serverless SQL can query Parquet/CSV in ADLS Gen2 directly — no ETL needed. Great for ad-hoc exploration.

        Azure Databricks is a managed Apache Spark platform optimised for collaboration between data engineers, data scientists, and analysts. It supports notebooks in Python, R, Scala, and SQL, and natively integrates with Delta Lake.

        • Built on Apache Spark — distributed data processing at scale
        • Delta Lake: open-source ACID storage layer on data lake
        • MLflow: built-in ML experiment tracking
        • Unity Catalog: unified data governance across workspaces
        ⚠ Exam tip: Databricks = Spark + Delta Lake + collaboration. If the exam mentions medallion architecture or Delta tables, think Databricks.

        Azure Stream Analytics is a fully managed real-time analytics service. It processes streaming data from Azure Event Hubs, IoT Hub, or Blob Storage using a SQL-like query language, and outputs results to various sinks.

        • Inputs: Event Hubs, IoT Hub, Blob Storage
        • Outputs: Power BI, Azure SQL DB, Blob, Cosmos DB, Event Hubs, more
        • Windowing functions: Tumbling, Hopping, Sliding, Session
        • Use cases: IoT telemetry, fraud detection, live dashboards
        ⚠ Exam tip: Stream Analytics uses a SQL-like language. Tumbling windows are non-overlapping fixed-size time windows.

        Power BI is Microsoft's self-service BI and visualisation platform. It has three components: Power BI Desktop (authoring), Power BI Service (cloud publishing and sharing), and Power BI Mobile (consumption on devices).

        • Desktop: build reports and data models locally
        • Service: publish, share, schedule refresh, create dashboards
        • Workspaces: collaboration areas for reports and datasets
        • DAX: formula language for calculated columns and measures
        • Direct Lake: query Delta tables in OneLake without import
        ⚠ Exam tip: A dashboard in Power BI Service is a single page of pinned visuals from multiple reports. A report is multi-page and built in Desktop.

        A star schema organises data warehouse tables into a central fact table (measures/events) surrounded by dimension tables (descriptive attributes for filtering and grouping). It is denormalised for query performance.

        • Fact table: numeric measures (sales amount, quantity), foreign keys to dimensions
        • Dimension table: descriptive attributes (product name, customer city, date)
        • SCD Type 1: overwrite old value — no history
        • SCD Type 2: add new row with new value — full history preserved
        ⚠ Exam tip: Power BI works best with a star schema. Snowflake schema (normalised dimensions) is supported but adds query complexity.
